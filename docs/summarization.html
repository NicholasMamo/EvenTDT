7. Summarization¶

Topic Detection and Tracking (TDT) approaches are normally accompanied by a summarization module. Whereas the TDT technique tells us that something happened, summarization explains what happened.

Summarization takes the machine-understandable data and creates human-understandable information. This package provides basic summarization resources, algorithms to summarize documents and functionality to build timelines.

The Summary Object¶

The Summary object is a special class that represents summaries. Essentially, this class is a group of Document, which can represent sentences, tweets and other text. Moreover, it includes additional, summary-related functionality and it can accept additional attributes.

The point of the Summary object is that it retains the original Document instances. Therefore a Summary instance is not a simple string. It can be represented or manipulated (such as by using a Cleaner) as the application necessitates by having access to its original components.

class Summary(documents=None, *args, **kwargs)¶

A summary is essentially a list of Document. However, summaries may also have other attributes.

The point of the summary object is to encapsulate the list of Document instances that make it up, while retaining the original text. The list of Document can be accessed using the documents property, which means that Document instances can be added or removed dynamically.

In addition, this class provides the __str__() special function, which concatenates its documents to form a textual summary. If the documents contain a timestamp attribute, it automatically sorts the documents in chronological order.

Algorithms¶

Summarization algorithms receive documents and create a summary having the given length. The specifics of how they create summaries change from one algorithm to the other, but the goal does not.

The SummarizationAlgorithm represents this goal. It is a base class that dictates the minimum functionality of any summarization algorithm: what input they receive and what output they return. In all cases, the main functionality goes through the summarize() function.

class SummarizationAlgorithm¶

Summarization algorithms vary greatly and thus there is no general state. However, they must all implement the summarize() method. This method is the central part of any SummarizationAlgorithm. It receives a list of Document instances and a target length and creates a Summary.

abstract summarize(documents, length, *args, **kwargs)¶

Summarize the given list of Document instances.

Summarization algorithms may accept more parameters, but they must accept, at least:

  1. A list of Document instances, and

  2. The maximum length of the produced Summary.

They must also always return a Summary instance.

Note

The algorithm assumes that all Document instances are already unique sentences. Therefore it does not split them into sentences or manipulate them in any way. The returned Summary is a selection of the given Document instances.

Parameters

  • documents (list of Document) – The list of documents to summarize.

  • length (float) – The maximum length of the summary in characters. Summarization algorithms that inherit the SummarizationAlgorithm must ensure that the length of the Summary does not exceed this length. Summaries may be shorter, however. When considering the length of the Summary, the function does not apply any pre-processing. All it considers is the text of the original Document instances making up the Summary.

Returns

The summary of the documents.

Return type

Summary

Carbonell and Goldstein (1998)¶

The Maximal Marginal Relevance (MMR) algorithm takes in documents and a query and builds a summary. This summary has the ideal property of being relevant to the query and non-redundant. The algorithm follows a greedy approach.

Note

Implementation based on the algorithm presented in The Use of MMR, Diversity-Based Reranking for Reordering Documents and Producing Summaries by Carbonell and Goldstein (1998).

class MMR(l=0.5)¶

The Maximal Marginal Relevance (MMR) algorithm accepts documents and a query and builds a summary. The algorithm looks to build a summary that is simultaneously similar to the query and non-redundant.

Variables

l (float) – The \(\lambda\) parameter, which must be between 0 and 1. When \(\lambda\) is 1, only relevance to the query is considered. When \(\lambda\) is 0, only non-redundancy is considered. By default, the algorithm assigns equal weight to relevance and non-redundancy.

Mamo et al. (2019)¶

The Document Graph Summarizer (DGS) algorithm takes in documents and builds a summary. The algorithm is not greedy and constructs a document graph, which it then splits into communities. Considering each community to be a different facet, the algorithm picks documents from the largest communities. In this way, the algorithm can maximize precision by minimizing redundancy.

Note

Implementation based on the algorithm presented in ELD: Event TimeLine Detection—A Participant-Based Approach to Tracking Events by Mamo et al. (2019).

class DGS¶

The Document Graph Summarizer (DGS) is an algorithm that minimizes redundancy by splitting documents into communities. The algorithm receives documents and builds a summary from the largest communities to capture all facets.

Timelines¶

A timeline is a succession of Node. Each node stores information about what happened in that period of time. The timeline groups these nodes together.

class Timeline(node_type, expiry, min_similarity, max_time=600, nodes=None)¶

The timeline stores a list of nodes and provides functionality to convert them into summaries. More importantly, the timeline provides functionality to construct the timeline. This includes redundancy management.

Incoming documents are added automatically to the latest node if it has not expired. At any time, the timeline only ever has one non-expired, or active, node. If there are no active nodes, the incoming documents can either be absorbed by an expired node, or go into a new node.

Variables

  • nodes (Node) – The list of nodes in the timeline.

  • node_type (Node) – The type of nodes to create in the timeline.

  • expiry (float) – The time in seconds that it takes for a node to expire. Expired nodes do not automatically absorb documents. If the expiry is 0, new documents immediately join a new node unless they are absorbed.

  • min_similarity – The minimum similarity between incoming documents and a node to be absorbed by it. This value is inclusive.

  • max_time (float) – The maximum time in seconds to look back when deciding whether a node should absorb a new topic. The comparison is made with the node’s created_at instance variable. This value is inclusive.

add(timestamp=None, *args, **kwargs)¶

Add documents to a node. This function first tries to identify if there is an active node. If there isn’t, it tries to find a node that can absorb the documents. This process is performed in reverse. If it doesn’t, a new node is created.

All arguments and keyword arguments are passed on to the add() and similarity() methods.

Nodes¶

Timelines are made up of nodes. Each node stores documents that can be summarized. Moreover, nodes provide functionality to facilitate timeline tasks.

class Node(created_at)¶

Nodes are the basic component of a timeline. Timelines contain an ordered list of these nodes.

Nodes can store documents in different ways. For example, they can be stored as simple documents. Alternatively, they can be stored as clusters.

Nodes must offer certain functionality to facilitate the timeline construction.

Note

The inputs for the add() and similarity() function should be the same. This is necessary even if an input is used in add(), but not in similarity(). This is because the add() function passes all arguments and keyword arguments in the same way.
